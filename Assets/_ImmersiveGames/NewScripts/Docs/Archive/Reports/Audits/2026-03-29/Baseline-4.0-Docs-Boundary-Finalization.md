# Baseline 4.0 - Docs Boundary Finalization

## O que ainda estava misturado

- O blueprint ainda carregava fases de implementação, backlog e linguagem de rollout.
- O backlog auxiliar ainda podia ser lido como contrato de arquitetura por causa da forma de algumas tabelas e headings.
- O README e o Canon Index ainda não separavam com clareza suficiente a cadeia canônica, o alvo, o guardrail operacional e o índice de estado atual/histórico.

## O que foi separado

- O blueprint ficou restrito a:
  - backbone conceitual
  - backbone de runtime
  - arquitetura alvo por domínio
  - mapa de reaproveitamento como inventário de apoio
  - regras estruturais permanentes
- O documento de reorganização passou a ser backlog auxiliar subordinado, com sequenciamento explicitado como backlog e nao como canon.
- O README passou a declarar a cadeia documental por papel.
- O Canon Index passou a declarar que e estado atual/historico, nao autoridade canonica do Baseline 4.0.

## Autoridade final por documento

- `ADR-0044` - canon arquitetural
- `ADR-0043` - ancora de decisao
- `Blueprint-Baseline-4.0-Ideal-Architecture.md` - referencia alvo
- `Plan-Baseline-4.0-Execution-Guardrails.md` - formato operacional obrigatorio
- `Plan-Baseline-4.0-Reorganization.md` - backlog auxiliar
- `Baseline-4.0-Docs-Alignment-Audit.md` - registro de alinhamento e gaps
- `Baseline-4.0-Docs-Boundary-Finalization.md` - registro do fechamento da fronteira documental
- `Canon-Index.md` - indice de estado atual/historico

## Ambiguidades restantes

- Nenhuma ambiguidade nova foi introduzida nesta etapa.
- A unica leitura residual a manter e a separacao intencional entre docs canonicos e docs de estado atual/historico, agora declarada explicitamente no README e no Canon Index.
