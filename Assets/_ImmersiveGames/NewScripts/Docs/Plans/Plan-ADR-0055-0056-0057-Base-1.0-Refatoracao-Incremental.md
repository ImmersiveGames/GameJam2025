# Plan - Base 1.0: refatoracao incremental do sistema composto (ADR-0055 / ADR-0056 / ADR-0057)

## 1. Objetivo

Organizar a refatoracao do sistema composto `Base 1.0` como um conjunto coordenado de tres camadas:

1. baseline tecnico fino
2. `Session Integration`
3. camadas semanticas acima do baseline

O plano nao trata um dominio isolado. Ele prepara o sistema inteiro para evoluir coerentemente com a topologia arquitetural ja congelada.

## 2. Direcao Arquitetural

Alvo:

- baseline como executor tecnico fino
- `Session Integration` como seam explicito acima do baseline
- `GameplayRuntimeComposition`, `GameplaySessionFlow`, `Session Transition`, participation e phase composition como camadas semanticas
- emissor canonico unico de intencao para seams adjacentes
- separacao clara entre `contracts`, `runtime`, `compat` e `adapters`

Nao-alvos:

- workarounds transitrios
- bootstrap como lugar de costura por conveniencia
- preservar a estrutura atual por inercia
- expandir spawn/reset/`ActorRegistry` antes de estabilizar o shape

## 3. Fases de Implementacao

### F1 - Congelamento operacional da Base 1.0

Objetivo:

- alinhar naming, boundaries e leitura em tres camadas no plano operacional
- garantir coerencia entre `ADR-0055`, `ADR-0056` e `ADR-0057`

Motivacao:

- sem leitura composta, cada refatoracao volta a parecer um problema local

Areas/arquivos mais provaveis:

- `Docs/ADRs/ADR-0055*`
- `Docs/ADRs/ADR-0056*`
- `Docs/ADRs/ADR-0057*`
- `Docs/ADRs/README.md`
- `Docs/Modules/README.md`

Dependencias:

- ADRs congelados e aceitos

Risco principal:

- continuar executando mudancas como se baseline, integration e semantica fossem frentes independentes

Pronto quando:

- a leitura `Base 1.0` e a topologia em tres camadas estao documentadas como referencia operacional
- a cadeia de precedencia entre os tres ADRs esta clara
- os docs de indice apontam para a leitura composta, nao apenas para ADRs isolados

### F2 - Criacao/estrutura da area de Session Integration

Objetivo:

- criar a area propria de integracao semantica de sessao
- preparar a relocalizacao das bridges e translators hoje espalhados

Motivacao:

- sem seam explicito, a costura continua migrando para bootstraps e installers

Areas/arquivos mais provaveis:

- `Orchestration/GameplaySession/Integration/**`
- `GameplaySessionFlow`
- `GameplaySessionContextService.cs`
- `GameplayParticipationContracts.cs`
- `SceneFlowBootstrap.cs`
- `SceneFlowInputModeBridge.cs`

Dependencias:

- F1 concluida
- fronteira entre baseline e semantica acima estabilizada

Risco principal:

- criar a pasta/namespace nova, mas manter a costura primaria nos lugares antigos

Pronto quando:

- a area de `Session Integration` existe como modulo proprio
- contratos minimos e ownership da area estao definidos
- bridges antigos deixam de ser o lugar principal de integracao

### F3 - Emissor canonico de intencao

Objetivo:

- congelar quem emite intencao canonica para seams adjacentes
- resolver a ambiguidade de `InputModes`

Motivacao:

- dedupe operacional nao pode continuar mascarando ownership errado

Areas/arquivos mais provaveis:

- `InputModes`
- `InputModeCoordinator.cs`
- `SceneFlowInputModeBridge.cs`
- `GameLoopInputCommandBridge.cs`
- `GameRunEndedEventBridge.cs`
- seam de `Session Integration`

Dependencias:

- F2 concluida
- area de integracao ja identificavel como owner da traducao de intencao

Risco principal:

- manter multiplas fontes de request para a mesma rail operacional

Pronto quando:

- existe um emissor canonico de intencao por tipo de seam adjacente
- `InputModeCoordinator` volta a ser apenas aplicador/dedupe operacional
- bridges restantes sao adapters finos, nao owners concorrentes

### F4 - Enxugamento do baseline

Objetivo:

- tirar do baseline a costura semantica que nao pertence a ele
- reduzir bootstrap bloat

Motivacao:

- o baseline deve voltar a ser executor tecnico fino, nao centro semantico

Areas/arquivos mais provaveis:

- `SceneFlowBootstrap.cs`
- `NavigationBootstrap.cs`
- `GameLoopBootstrap.cs`
- `GameLoopInstaller.cs`
- `GlobalCompositionRoot.*`

Dependencias:

- F2 e F3 estabilizadas

Risco principal:

- quebrar a ordem canonica de startup ao remover costuras antes de mover os consumidores

Pronto quando:

- `SceneFlowBootstrap` encolheu para wiring tecnico
- `NavigationBootstrap` voltou a macro dispatch puro
- partes de `GameLoopBootstrap` deixam de carregar sync transversal sem dono claro

### F5 - Fatiamento de buckets e limpeza modular

Objetivo:

- separar `contracts`, `runtime`, `compat` e `adapters`
- reescopar namespaces residuais

Motivacao:

- buckets mistos escondem ownership e dificultam a evolucao do sistema

Areas/arquivos mais provaveis:

- `GameplaySessionContextService.cs`
- `GameplayParticipationContracts.cs`
- `LevelLifecycle/**`
- `PhaseDefinition/Runtime/**`
- `GameplaySessionFlow/**`

Dependencias:

- F2 e F4 estabilizadas

Risco principal:

- fragmentar antes de estabilizar os novos owners

Pronto quando:

- os buckets inflados foram fatiados por responsabilidade
- `LevelLifecycle` residual deixa de funcionar como namespace historico ativo
- compat e adapters deixam de morar no mesmo arquivo que contracts e runtime

### F6 - Reintegracao coordenada dos dominios adjacentes

Objetivo:

- reconectar `InputModes`, spawn, reset, `ActorRegistry` e seams adjacentes dentro do shape novo
- manter ownership semantico fora do baseline

Motivacao:

- os dominios operacionais precisam voltar a conversar com um seam claro, nao com bootstrap oportunista

Areas/arquivos mais provaveis:

- `InputModes`
- `ResetInterop`
- `WorldReset`
- `SceneReset`
- `ActorRegistry`
- spawn/materializacao
- adapters de `Session Integration`

Dependencias:

- F2, F3, F4 e F5 concluidas

Risco principal:

- devolver ao baseline responsabilidade semantica por causa da integracao operacional

Pronto quando:

- os dominios adjacentes consomem a intencao canonica do seam
- spawn/reset/`ActorRegistry` continuam operacionais, nao semanticos
- a integracao e explicitamente mediada pela area de `Session Integration`

### F7 - Preparacao do sistema para os proximos eixos

Objetivo:

- deixar `Base 1.0` pronta para evolucao de actors, binders/interacoes, session transition e novos blocos semanticos

Motivacao:

- a arquitetura deve crescer a partir de fronteiras estaveis, nao de novos remendos

Areas/arquivos mais provaveis:

- `Docs/Modules/**`
- `Docs/ADRs/**`
- `GameplayRuntimeComposition`
- futuros blocos de actors e binders

Dependencias:

- F1 a F6 concluidas e estabilizadas

Risco principal:

- abrir novos eixos antes de o sistema composto estar realmente estavel

Pronto quando:

- a expansao futura usa o mesmo vocabulrio de camadas e ownership
- novos blocos entram acima do baseline sem reintroduzir costura por conveniencia
- o plano futuro pode ser derivado da topologia de `Base 1.0` sem ambiguidade

## 4. Sequencia Recomendada

1. congelar `Base 1.0` como leitura operacional
2. criar e estabilizar `Session Integration`
3. definir emissor canonico de intencao
4. enxugar o baseline
5. fatiar buckets e limpar namespaces residuais
6. reintegrar dominios adjacentes
7. preparar os proximos eixos

## 5. Resumo de Controle

O plano so esta pronto quando:

- baseline, integracao e semantica acima sao lidos como um sistema unico
- `Session Integration` e o unico lugar canonico de traducao semantica -> operacional
- bootstraps deixam de ser ponto de costura
- expansoes futuras nao dependem de workaround transitorio
