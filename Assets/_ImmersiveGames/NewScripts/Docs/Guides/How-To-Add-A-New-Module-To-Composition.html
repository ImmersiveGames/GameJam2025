Como adicionar um módulo sem quebrar o modelo

O contrato atual em `NewScripts` separa duas fases globais: installers primeiro e bootstraps/composers depois. O módulo novo deve entrar nesse modelo por descriptor explícito, sem reflection, sem auto-discovery e sem wiring escondido no root.

Installerregistro pre-runtime apenas.
Bootstrapcomposição runtime apenas, quando necessário.
Descriptordeclara módulo, dependências e entry points.
Rootagrega descriptors e executa o grafo.
Installer only Installer + Bootstrap Arquivos requeridos

Quando um módulo precisa só de installer

Use apenas installer quando o módulo só precisa registrar serviços, contratos, bridges finos ou helpers pre-runtime. Não crie bootstrap artificial se não existir composição runtime real.

Exemplo semelhante
WorldReset: registra WorldResetService, IWorldResetCommands, SceneFlowWorldResetDriver e IWorldResetRequestService. Não há bootstrap separado.
Sinal de que basta installer
  • não existe wiring operacional posterior;
  • não há bridges/coordinators para ativar depois;
  • o módulo não depende de outros serviços runtime para completar o boot.

Quando um módulo precisa de installer + bootstrap

Use as duas fases quando o módulo precisa registrar primeiro e depois compor/ativar runtime operacional. O bootstrap deve usar apenas o DI já preenchido.

GameLoopinstaller registra contratos e services; bootstrap conecta sync runtime.
SceneFlowinstaller registra policies, resolvers e services; bootstrap monta o stack de transição.
Navigationinstaller registra catálogo; bootstrap ativa coordinators e bridges.
LevelFlowinstaller registra prereqs; bootstrap faz wiring dependente de runtime.

Regra prática
Se o módulo não tiver uma fase runtime real para compor, não invente um bootstrap só para “fechar padrão”.

Responsabilidades mínimas

Installer faz
  • registra serviços, contratos, providers, factories e configs
  • valida pré-requisitos pre-runtime
  • não depende de bootstrap
  • não compõe runtime operacional
Bootstrap faz
  • compoe runtime operacional
  • ativa bridges, coordinators e orchestrators
  • usa apenas serviços já registrados
  • não faz registro pre-runtime

Declarando dependências

O descriptor deve declarar explicitamente as dependências por fase:

  • ModuleId
  • InstallerDependencies
  • BootstrapDependencies
  • referência ao installer
  • referência ao bootstrap, quando existir
Exemplo de descriptor
public static class WorldResetCompositionDescriptor
{
    public static ICompositionModuleDescriptor Descriptor { get; } =
        new CompositionModuleDescriptor(
            moduleId: "WorldReset",
            installerDependencies: Array.Empty<string>(),
            bootstrapDependencies: Array.Empty<string>(),
            installer: _ => WorldResetInstaller.Install(_),
            bootstrap: null);
}
Exemplo com duas fases
public static class NavigationCompositionDescriptor
{
    public static ICompositionModuleDescriptor Descriptor { get; } =
        new CompositionModuleDescriptor(
            moduleId: "Navigation",
            installerDependencies: Array.Empty<string>(),
            bootstrapDependencies: new[] { "SceneFlow" },
            installer: bootstrapConfig => NavigationInstaller.Install(bootstrapConfig),
            bootstrap: bootstrapConfig => NavigationBootstrap.ComposeRuntime(bootstrapConfig));
}
Regra obrigatória
Installer não depende de bootstrap. Bootstrap pode depender de bootstrap e de serviços já registrados.

Registrando o módulo no root

O root não deve conhecer wiring interno do módulo. Ele apenas agrega o descriptor no grafo global.

steps.Add(CompositionPipelineStep.FromDescriptor(WorldResetCompositionDescriptor.Descriptor));
O que muda
O root deixa de declarar serviços internos do módulo e passa a ser apenas agregador + executor do grafo + orquestrador das fases globais.

Fail-fast

O pipeline deve falhar cedo quando houver:

  • dependência ausente
  • dependência inválida
  • ID duplicado
  • ciclo
  • dependência declarada na fase errada
Motivo
O modelo modular só fica previsível quando a ordem é derivada de dependências explícitas e não de ordem manual espalhada.

Checklist prático

  • Criar XxxInstaller
  • Criar XxxCompositionDescriptor
  • Criar XxxBootstrap somente se houver runtime real
  • Declarar dependências explícitas por fase
  • Adicionar o descriptor ao grafo global
  • Confirmar que o installer não depende de bootstrap
  • Confirmar que o root não carrega wiring interno do módulo

Erros comuns

Misturar fasescolocar composição runtime dentro do installer.
Root gordomanter wiring interno no GlobalCompositionRoot.
Dependência erradadepender de bootstrap na fase de installer.
Bootstrap artificialcriar bootstrap sem necessidade real.
Prioridade manualusar número solto como source-of-truth de ordem.

Resumo final

Para adicionar um módulo novo com segurança: descreva o módulo, registre primeiro, componha depois só se houver runtime real, e deixe o root apenas agregar o descriptor e executar o grafo.
